# Характеристика проекта

Контрольная работа №4 является результатом выполнения практических заданий №19–23. В рамках работы были выполнены задания по PostgreSQL, MongoDB, Redis, балансировке нагрузки, Docker и Docker Compose.

В ходе работы были реализованы:

- API для управления пользователями;
- подключение PostgreSQL;
- подключение MongoDB;
- CRUD-операции для пользователей;
- хранение данных пользователей в базе данных;
- кэширование GET-запросов через Redis;
- очистка кэша при изменении данных;
- несколько backend-серверов на Node.js;
- балансировка нагрузки через Nginx;
- альтернативная балансировка через HAProxy;
- настройки отказоустойчивости `max_fails` и `fail_timeout`;
- контейнеризация приложения с Docker;
- запуск проекта через Docker Compose;
- объединение сервисов в одну Docker-сеть;
- проверка распределения запросов между backend-серверами.

# Функциональность

## API пользователей

В проекте реализованы маршруты для управления пользователями.

Реализовано:

- создание пользователя;
- получение списка пользователей;
- получение пользователя по ID;
- обновление информации о пользователе;
- удаление пользователя.

---

## PostgreSQL

В проект добавлена работа с реляционной базой данных PostgreSQL.

Реализовано:

- подключение Node.js-приложения к PostgreSQL;
- создание таблицы пользователей;
- хранение пользователей в PostgreSQL;
- выполнение CRUD-операций через API;
- создание нового пользователя;
- получение списка пользователей;
- получение пользователя по ID;
- обновление пользователя;
- удаление пользователя;
- связь каждого маршрута с базой данных PostgreSQL.

---

## MongoDB

В проект добавлена работа с NoSQL-базой данных MongoDB.

Реализовано:

- подключение Node.js-приложения к MongoDB;
- использование Mongoose;
- создание модели пользователя;
- хранение пользователей в коллекции MongoDB;
- создание документов;
- получение документов;
- обновление документов;
- удаление документов;
- связь каждого маршрута API с MongoDB.

---

## Redis-кэширование

В проект добавлено кэширование с использованием Redis.

Redis используется для ускорения повторных запросов. При первом обращении данные получаются с сервера и сохраняются в Redis. При повторном обращении эти же данные возвращаются уже из кэша.

Реализовано:

- подключение Redis;
- middleware для чтения данных из кэша;
- сохранение данных в Redis;
- кэширование списка пользователей;
- кэширование пользователя по ID;
- кэширование списка товаров;
- кэширование товара по ID;
- очистка кэша при изменении данных.

---

## Балансировка нагрузки

В проект добавлена тестовая система балансировки нагрузки для веб-приложения.

Реализовано:

- несколько backend-серверов;
- запуск backend-серверов на разных портах;
- настройка Nginx как балансировщика нагрузки;
- распределение запросов между backend-серверами;
- резервный backend-сервер;
- настройка отказоустойчивости через `max_fails` и `fail_timeout`;
- проверка работы балансировки через повторные HTTP-запросы;
- альтернативный пример балансировки через HAProxy.

---

## Nginx

Nginx используется как основной балансировщик нагрузки.

Он принимает входящие HTTP-запросы и перенаправляет их на один из backend-серверов.

Реализовано:

- блок `upstream`;
- список backend-серверов;
- распределение запросов между серверами;
- резервный backend-сервер;
- параметры отказоустойчивости.

---

## HAProxy

HAProxy добавлен как альтернативный пример балансировки нагрузки.

Реализовано:

- frontend для приема HTTP-запросов;
- backend-группа серверов;
- алгоритм балансировки `roundrobin`;
- проверка доступности серверов через `check`;
- резервный сервер через `backup`.

---

## Docker и Docker Compose

Проект контейнеризирован с использованием Docker.

Реализовано:

- Dockerfile для backend-сервиса;
- `docker-compose.yml` для запуска всех сервисов;
- отдельные контейнеры для backend-серверов;
- контейнер Nginx;
- контейнер HAProxy;
- контейнер Redis;
- контейнеры баз данных;
- общая Docker-сеть для взаимодействия сервисов;
- запуск всего проекта одной командой.
